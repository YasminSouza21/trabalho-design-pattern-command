Contexto histórico
Como o Command surgiu
A ideia não nasceu do nada — ela é produto de décadas de evolução no pensamento sobre software orientado a objetos.
Anos 1970
Origens na programação funcional
A ideia de tratar funções como cidadãos de primeira classe, passando-as como argumentos, vem de linguagens como Lisp. Isso plantou a semente do Command.
Anos 1980
Smalltalk e as GUIs
Com as interfaces gráficas, ficou claro que botões precisavam disparar ações sem acoplamento direto. O Smalltalk explorou padrões que anteciparam o Command.
1994
Gang of Four formaliza o padrão
Publicação do famoso livro "Design Patterns", documentando formalmente o Command como um dos 23 padrões clássicos da engenharia de software.
Hoje
Modernidade (Go, Rust, TS)
Em linguagens como Go, a combinação de interfaces simples com structs torna a implementação do Command extremamente limpa, sem excesso de classes.
Mundo real
A analogia do restaurante
O Command funciona exatamente como o fluxo de pedidos de um restaurante. Nenhuma parte precisa conhecer os detalhes das outras.
No restaurante
Cliente
Faz o pedido ao garçom sem saber cozinhar
Garçom
Anota o pedido no papel e leva pra cozinha
Papel
Contém as informações: prato, mesa, etc.
Cozinha
Recebe o papel e prepara a comida
No código
Cliente
Configura comandos e associa receptores
Invocador
Armazena o comando e chama Execute()
Comando
Guarda a operação, parâmetros e o receptor
Receptor
Executa a lógica real (a regra de negócio)

Ponto chave: O garçom não sabe cozinhar. A cozinha não fala com o cliente. O papel do pedido é a ponte — assim como o objeto Command conecta Invocador e Receptor mantendo o desacoplamento.

Aplicações
Casos de uso reais
O Command aparece em mais lugares do que parece. Reconhecê-los é o primeiro passo para saber quando aplicar.
Undo / Redo
Editores de texto e Photoshop. Cada ação é um comando na pilha. Ctrl+Z apenas chama o Undo().
Filas Assíncronas
Sistemas que enfileiram comandos para execução posterior ou distribuída em background.
Replay de Jogos
Cada botão pressionado vira um comando. O replay apenas re-executa a lista de comandos gravada.
Transações (BD)
Comandos encapsulam commits e rollbacks, garantindo que operações em lote não falhem no meio.
Estrutura
Os 5 Componentes
Cada peça tem uma responsabilidade única e bem definida. Juntas, elas promovem a essência do padrão.
01
Interface Command interface
Declara o contrato. Em Go, geralmente possui apenas Execute() e Undo(). O Invocador só conhece essa interface, ignorando o que rola por trás.
02
Comando Concreto struct
Implementa a interface Command. Ele guarda uma referência para o receptor e os parâmetros necessários para a ação (ex: EscreverComando).
03
Receptor (Receiver) struct
A regra de negócio real (ex: o próprio Editor). Ele sabe como executar o trabalho, mas não sabe que o padrão Command existe no sistema.
04
Invocador (Invoker) struct
Responsável por disparar a requisição (ex: um botão de interface). Ele segura os Comandos numa pilha e chama Execute() quando necessário.
05
Cliente main()
Monta todo o "Lego". Cria receptores, instancia os comandos passando o receptor pra eles, e entrega os comandos prontos para os invocadores.

Avaliação
Quando usar e quando evitar
O padrão não é uma solução universal. Avalie os prós e contras antes de encher seu código de structs.
Use quando...

Precisar de undo/redo com histórico de ações.

Quiser enfileirar operações para execução assíncrona.

Diferentes partes do sistema (menu, atalho) disparam a mesma ação.

Evite quando...

A operação for muito simples (uma chamada de função direta resolve).

Nenhuma ação precisar ser desfeita ou enfileirada no futuro.

A complexidade de criar 10 arquivos novos não compensa o problema.
